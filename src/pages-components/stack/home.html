spa-loader-content-html

Plugin vanilla JavaScript para SPAs — v3

spa-loader-content-html es un plugin vanilla JavaScript diseñado para crear aplicaciones de una sola página (SPA) sin frameworks. Carga fragmentos HTML dinámicamente mediante fetch(), inyectándolos en contenedores del DOM definidos por rutas configuradas en un manifiesto ligero.

La versión 3 introduce lazy loading de módulos de ruta mediante import() dinámico, soporte nativo para la View Transitions API, integración con Markdown Shiki para resaltar código fuente en cada vista, reescritura automática de URLs en HTML inyectado, metadatos dinámicos por ruta (título, favicon, cabecera), navegación con History API, caché de módulos por ruta y un sistema de eventos personalizados para desacoplar el loader visual de la lógica de navegación.

Características Principales

🚀

Lazy Loading de Rutas

Los módulos de ruta se importan dinámicamente con import() solo cuando el usuario navega a esa ruta. Un manifiesto ligero (routeManifest) registra cada ruta con id, path y file, sin importar módulos en el arranque.

🎬

View Transitions API

Cuando el navegador soporta document.startViewTransition(), el plugin envuelve automáticamente el renderizado de la nueva ruta en una transición nativa, logrando animaciones fluidas entre páginas sin configuración adicional.

📋

Route Manifest

El routeManifest es un array de entradas { id, path, file }. El plugin resuelve la ruta activa comparando window.location.pathname normalizado con cada path, y carga el módulo correspondiente bajo demanda.

🔃

Inyección de HTML por Componente

Cada ruta define un objeto components donde la clave es el id del contenedor en el DOM y el valor es la URL del fragmento HTML. El plugin hace fetch() de cada fragmento y lo inyecta en su contenedor de forma secuencial.

🎨

CSS y JS por Ruta

Cada ruta puede declarar sus propios styles (array de { href }) y scripts (array de { src, isModule? }). El plugin los inyecta dinámicamente en el <head> al navegar a esa ruta.

⚡

Caché y Eventos Personalizados

Los módulos de ruta se cachean en un Map interno para evitar importaciones redundantes. El plugin emite eventos spa:route-loaded, spa:first-route-loaded y spa:route-load-error para desacoplar el loader visual de la navegación SPA.

✨

Markdown Shiki

Cada ruta puede declarar un array MarkdownShikiHtml con entradas { url, target }. Tras cargar la vista, el plugin hace fetch() de los bloques HTML generados con Shiki y los inyecta en contenedores [data-shiki="..."]. Regenera el resaltado con pnpm code-highlight.

🔗

Reescritura de URLs en HTML

Al inyectar fragmentos HTML, el plugin reescribe automáticamente las rutas relativas de src, href, poster y srcset respecto a la URL del fragmento origen, evitando enlaces rotos cuando los componentes referencian assets en su propio directorio.

🧭

Navegación con History API

Intercepta clics en enlaces internos con data-id y actualiza la URL con history.pushState() sin recargar la página. El evento popstate restaura la ruta, el favicon y el contenido al usar los botones atrás/adelante del navegador.

🏷️

Metadatos Dinámicos por Ruta

Cada ruta puede definir pageTitle, headerTitle y favicon. El plugin actualiza el <title>, el encabezado del layout y el icono de la pestaña al navegar, con cache-busting en el favicon para evitar parpadeos entre vistas.

🚫

Ruta 404 Integrada

Si la URL no coincide con ninguna entrada del manifest, el plugin carga automáticamente la ruta 404NotFoundPage configurada en el manifiesto. Funciona en carga inicial, clics internos y navegación popstate, evitando pantallas en blanco o loaders bloqueados.

⏳

Loader Inicial Desacoplado

El efecto effect-loading-page.js escucha spa:first-route-loaded y spa:route-load-error para ocultar el loader solo cuando la primera ruta SPA termina de cargar (o tras un timeout de fallback), manteniendo la UX fluida desde el arranque.

Pipeline de Desarrollo

El proyecto separa el código fuente (src/) del artefacto servible (app/) y del build de producción minificado (dist/). Gulp orquesta la copia, compilación SCSS, generación Shiki y minificación; los servidores Express añaden fallback SPA, live reload y soporte PHP en desarrollo.

📁

Gulp — src → app

La tarea copyAll copia componentes, páginas, rutas, plugins, scripts y efectos desde src/ a app/, y compila globals.scss y los SCSS de cada página a CSS con sourcemaps.

🖍️

generate-markdown-shiki

Tras compilar estilos, generateShiki lee las rutas del SPA, deduce los archivos fuente por convención de nombres (-html, -css, -scss, -js) y genera bloques HTML resaltados con Shiki (tema dark-plus) en app/markdown-shiki/.

👀

Watch en Tiempo Real

gulp dev ejecuta copyAll y observa cambios en src/. Al modificar SCSS, recompila estilos, regenera Shiki y sincroniza markdown-shiki. Soporta polling vía CHOKIDAR_USEPOLLING para entornos Linux con límites de inotify.

🌐

Dev Server + BrowserSync

pnpm dev lanza Gulp watch y un servidor Express con fallback SPA, archivos estáticos bajo la base del proyecto, soporte php-cgi y proxy BrowserSync con live reload en app/ e index.html (puerto configurable con DEV_SERVER_PORT).

📦

Build de Producción

pnpm build limpia dist/ y app/, regenera app/ desde src/ y minifica HTML, CSS y JS con htmlmin, clean-css y terser hacia dist/, listo para despliegue.

🔍

Preview Server

pnpm preview sirve el contenido de dist/ con Express, fallback SPA y la misma base de rutas que en desarrollo. Permite validar el build minificado antes de publicar (puerto PREVIEW_SERVER_PORT, por defecto 4173).
